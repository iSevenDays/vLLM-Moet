# AGENTS.md — the commit contract

Multiple agents and humans work on this project at the same time. They
often use the same checkout. Three times, work was lost because the
distribution patch was edited carelessly. Read this document before
your first commit. Each rule below comes from a real incident.

## The active lineage

The active deployment lineage is **v0.25.1** (sm_89 / Ada). The
Dockerfile applies `patch/vllm-moet-v0.25.1.patch` to the upstream
`vllm/vllm-openai:v0.25.1` image at build time. The patch and its
fingerprints (`patch/FILES-v0.25.1.txt` and `patch/SOURCE-v0.25.1.txt`)
are the source of truth for this lineage. Edit the patch directly. Use
the roundtrip-SHA discipline below. For vLLM source reference
(jump-to-def, "how does upstream do X"), read `vllm-project/vllm` on
`main`.

### The patch-direct procedure

1. **Stage the source.** Extract the current file body from the patch
   into a working copy (for example, `staging/<topic>/…`). Modify the
   file. Validate the syntax:

   ```bash
   python3 -c "import ast; ast.parse(open(<file>).read())"
   ```

   Keep the pristine extraction as `<file>.orig` for diff review.

2. **Locate the file's hunk** in `patch/vllm-moet-v0.25.1.patch`. New
   files use this form:

   ```
   new file mode 100644
   index 0000000..<sha>
   --- /dev/null
   +++ b/<file>
   @@ -0,0 +1,N @@
   ```

   Modified files use the standard unified diff form. Update hunks in
   place. Preserve context lines.

3. **Rewrite the body.** Replace the body with the modified file
   content. Prefix each line with `+`. Update the count in
   `@@ -0,0 +1,N @@`. Update the `index` blob SHA. Use this Python
   expression for the abbreviated git blob hash:

   ```python
   hashlib.sha1(b"blob " + str(len(content_bytes)).encode()
                + b"\0" + content_bytes).hexdigest()[:7]
   ```

4. **Roundtrip-verify.** Extract the body from the rewritten patch.
   Confirm its git blob SHA equals the modified file's git blob SHA. If
   the SHAs do not match, do not commit.

5. **Validate.** Run `git apply --check` on a synthetic tree. This
   confirms the patch is well-formed. For GPU-relevant changes, run the
   relevant suites (`tools/test_moe_w2_forward.py`,
   `tools/test_store_backends.py`, the three-tier tests). Put the
   results in the commit message.

6. **Commit.** Name the file(s) touched in the commit body.

The v0.25.1 fingerprints (`FILES-v0.25.1.txt` and
`SOURCE-v0.25.1.txt`) are regenerated by
`python3 tools/check_patch_files.py`. The byte-verify pass is skipped
because no fork clone is available on the active boxes.

## Legacy: the v0.24.0 fork-branch workflow (retired 2026-07-19)

The retired v0.24.0 lineage was generated from a fork branch
(`moet-v0.24.0` at `kacper-daftcode/vllm`). That branch covers only
the sm_120 (Blackwell) path. It does not cover sm_89. The active
lineage does not use it. If you ever need to revive v0.24.0, the
regen-from-branch tooling still works:

```bash
VLLM_MOET_FORK=/path/to/vllm-v0.24.0-clone \
  python3 tools/check_patch_files.py --update v0.24.0
```

## Where a change goes

| change | commit where | must also update |
|---|---|---|
| vLLM runtime code (`moe_w2_*`, loaders, runner, attention, …) | the v0.25.1 patch directly (procedure above) | commit the rewritten patch; `FILES-v0.25.1.txt` is auto-derived |
| SASS kernels / cubins | `kernels/` | add a `kernels/MANIFEST.md` row (generator + validation status) |
| serve configs | `bench/recipes/` | `bench/models.yaml`, `bench/matrix.yaml`; run `bench/runner/lint.py` |
| bench results | `bench/results/<release>/` | `bench/runner/render.py` (the README table and per-release report are generated; do not hand-edit the marked README block) |
| docs | `docs/`, README outside the generated block | — |
| session notes / handoffs / experiment scraps | `internal/` (gitignored, stays local) | do not commit into `docs/` |

Do not commit wheels (`patch/*.whl` is ignored), checkpoints, expert
packs, or smoke results (`bench/results/smoke/`).

## Concurrency — several agents, one checkout

- Run `git status` before you start. Dirty paths you did not create
  belong to another live session. Leave them alone. Do not run
  `git add -A`, `git commit -a`, `git stash`, `git checkout --`, or
  `git reset` over someone else's files.
- Stage explicit paths only: `git add <file> <file> …`.
- `main` is a shared trunk. Do not amend commits you did not just
  create. Do not rebase. Do not force-push. Do not rewrite history.
- The `patch/` trio (patch, `FILES-v0.25.1.txt`, `SOURCE-v0.25.1.txt`)
  changes only via the patch-direct procedure above (or the legacy
  `--update` for v0.24.0). If your commit touches any of them for
  another reason, the change is wrong.
- A pre-commit hook in dev-box checkouts runs the patch guard when
  `patch/` is staged. Do not bypass it with `--no-verify`.
- Commit identity: the global git config on this box holds the repo
  owner's identity. Use plain `git commit`. Do not pass
  `-c user.name=…` / `-c user.email=…` overrides; they attribute
  the commit to the wrong identity. (A prior workflow that overrode
  the identity per command produced mis-attributed commits; that
  workflow is gone.)

  Upstream PRs use the same identity + DCO. See
  `internal/UPSTREAM_PRS.md`. Do not edit git config.

## Pre-push checklist (mirrors CI `bench-lint`)

```bash
python3 tools/check_patch_files.py       # patch <-> FILES <-> SOURCE
python3 bench/runner/lint.py             # recipes/boxes/suites/results schemas
python3 bench/runner/render.py --check   # README table == committed results
```

Run `python3 docker/serve_recipe.py <recipe> --print` if you touched
recipes or the launcher.

## The incidents these rules come from

These incidents happened in the v0.24.0 era. At that time, the patch
was regenerated from a fork branch. Hand edits were silently dropped
by the next regen. The v0.25.1 lineage has no regen. The patch is the
source. The failure mode is now a careless hand edit, not a regen that
erases work. The discipline is the same: roundtrip-verify, name the
files, review the diff.

- **Kimi-K2.7 bring-up** — landed in the patch without its commits
  reaching the fork branch. The next regeneration erased it. It was
  caught by hand and folded back in `81c1b34`.
- **DSpark backport** — `ad7f29a` was regenerated from a branch that
  lacked the DSpark line. The line was silently dropped. The fix was
  to merge DSpark into the generating branch and regenerate the union
  (`8aff1b7`).
- **Stream-build hunks** — lost to a merge-side resolution inside the
  patch file. Restored by hand in `8f50e57`. Hunk-level losses inside
  an unchanged file set are invisible to `FILES.txt`. The
  `SOURCE.txt` byte-verification catches that class.

**A failing guard means work would be lost.** Redo the patch-direct
edit with roundtrip verification. Do not force the patch, `FILES`, or
`SOURCE` into agreement without verification.

## Commit style

Match `git log`. Use a declarative subject line: what ships or what
changed. Do not use prefixes. In the body, give the why and the
evidence (measured numbers, test verdicts). For v0.25.1, name the
file(s) the patch-direct edit touched.
